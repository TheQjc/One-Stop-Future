# 校园一站式信息平台（就业 / 考研 / 出国留学）正式需求清单

> **Active baseline note:** This is the active formal requirements baseline for the regenerated study-career-platform scope. Downstream phase designs, implementation plans, and execution records should trace back to this document unless a later approved requirements revision supersedes it.

## 1. 文档目的

本文档基于《校园一站式信息平台_软件需求说明书》整理当前项目的正式需求清单，作为后续架构、实施计划、编码和验收的统一依据。

本文档覆盖全局范围，但不表示全部内容同时进入当前实施阶段。当前实现优先级仍以设计说明中定义的阶段 A 为主。

## 2. 需求来源与优先级规则

### 2.1 唯一来源

本清单的需求来源按优先级从高到低为：

1. `docs/校园一站式信息平台_软件需求说明书.docx`
2. `docs/superpowers/specs/2026-04-15-study-career-platform-regeneration-design.md`

### 2.2 范围解释规则

- 需求说明书中明确写出的能力，按正式功能需求记录。
- 由说明书中的角色、接口、状态机、性能或部署章节稳定推出的内容，按约束需求记录。
- 设计说明中已经确认的范围拆分与阶段边界，按当前实施边界记录。

## 3. 角色定义

### 3.1 角色清单

- 游客：未注册或未登录用户，只能访问公开内容。
- 普通用户：通过手机号注册登录的用户，可使用前台完整基础能力，并可提交学号认证。
- 已认证用户：通过学号认证审核的用户，在普通用户基础上带有认证标识和更高内容可信度。
- 管理员：具备后台审核、管理和运营查看权限的用户。

### 3.2 当前优先级

- P0：平台基础域、认证域、首页聚合、个人中心、通知、认证审核、最小后台骨架
- P1：社区、岗位聚合、资料库、统一搜索
- P2：路径推荐、院校对比、完整数据看板、扩展运营能力

## 4. 全局功能需求

### 4.1 认证与账户

- `FR-AUTH-001`
  名称：手机号注册
  描述：系统应支持游客使用手机号和验证码完成注册。
  类型：显式需求
  适用角色：游客
  优先级：P0
  来源：SRS UC-01

- `FR-AUTH-002`
  名称：手机号登录
  描述：系统应支持已注册用户使用手机号和验证码完成登录。
  类型：显式需求
  适用角色：普通用户 / 已认证用户 / 管理员
  优先级：P0
  来源：SRS UC-02

- `FR-AUTH-003`
  名称：注册后直接进入登录态
  描述：系统应在注册成功后直接建立登录态并返回首页所需的用户上下文。
  类型：显式需求
  适用角色：游客
  优先级：P0
  来源：SRS UC-01

- `FR-AUTH-004`
  名称：账号状态控制
  描述：系统应支持账号封禁状态校验；被封禁用户登录时应收到明确提示。
  类型：显式需求
  适用角色：普通用户 / 已认证用户 / 管理员
  优先级：P1
  来源：SRS UC-02, 管理后台说明

- `FR-AUTH-005`
  名称：游客写操作拦截
  描述：系统应禁止游客执行发帖、评论、点赞、收藏、上传资料等写操作，并引导其登录。
  类型：显式需求
  适用角色：游客
  优先级：P0
  来源：SRS 角色权限汇总

## 5. 约束需求

### 5.1 接口与认证

- `CR-AUTH-001`
  名称：JWT 认证
  描述：系统应使用 JWT 作为登录凭证，并用于保护写接口和登录后接口。
  类型：显式约束
  优先级：P0
  来源：SRS 定义, 接口章节

- `CR-AUTH-002`
  名称：验证码格式
  描述：手机号验证码应为 6 位数字，有效期 5 分钟。
  类型：显式约束
  优先级：P0
  来源：SRS 输入要求, UC-01, UC-02

- `CR-AUTH-003`
  名称：手机号格式
  描述：登录和注册使用的手机号应为 11 位数字。
  类型：显式约束
  优先级：P0
  来源：SRS 输入要求

- `CR-AUTH-004`
  名称：Token 过期处理
  描述：系统在 Token 过期后应清理登录态并引导用户重新登录。
  类型：显式约束
  优先级：P0
  来源：SRS 故障处理

### 5.2 权限与角色

- `CR-PERM-001`
  名称：后台独立鉴权
  描述：管理后台能力必须经过独立权限校验，不能只依赖前端隐藏入口。
  类型：设计约束
  优先级：P0
  来源：设计说明, SRS 安全性

- `CR-PERM-002`
  名称：游客公开范围
  描述：游客只能访问公开内容、公开看板和公开搜索结果。
  类型：显式约束
  优先级：P0
  来源：SRS 角色权限汇总

- `CR-PERM-003`
  名称：已认证用户可信度提升
  描述：已认证用户发布的经验内容在后续推荐场景中应获得更高可信度或展示权重。
  类型：显式约束
  优先级：P1
  来源：SRS 角色定义

  描述：收藏记录应支持通过类型字段区分帖子、岗位和资料。

  来源：SRS 数据实体

- `CR-DATA-004`
  名称：评论层级

- `CR-DATA-005`

### 5.4 搜索、缓存与存储

- `CR-SEARCH-001`
  名称：Elasticsearch 主检索
  描述：系统应以 Elasticsearch 作为中文全文检索主实现。
  类型：显式约束
  优先级：P1
  来源：SRS 定义, 接口章节

- `CR-SEARCH-002`
  名称：搜索降级
  描述：当 Elasticsearch 不可用时，系统应降级为数据库模糊查询。
  类型：显式约束
  优先级：P1
  来源：SRS 故障处理

- `CR-CACHE-001`
  名称：Redis 排行与缓存
  描述：系统应使用 Redis 支撑缓存和排行榜相关能力。
  类型：显式约束

- `CR-CACHE-002`
  名称：缓存降级
  描述：当 Redis 不可用时，系统应降级为数据库直查，并允许排行榜暂不可用。
  类型：显式约束
  优先级：P1
  来源：SRS 故障处理
