# 校园一站式信息平台 Phase Q 社区经验帖结构化增强设计说明
## 1. 文档目标

本文档用于确认 `FR-COMMUNITY-002 经验帖结构化字段` 的首期实现边界。
本期目标不是引入新的帖子系统，也不是为就业 / 考研 / 留学分别设计完全不同的经验模板，而是在现有社区帖子模型上增加一层可选的结构化增强，使平台可以：

- 支持用户把普通帖子标记为经验帖
- 为经验帖补充少量稳定的结构化字段
- 在社区列表与详情页中更清晰地展示经验帖关键信息
- 在首页推荐区与发现页中给予经验帖更高展示权重

从而用最小改动补齐经验内容沉淀与推荐曝光能力。

## 2. 当前结论

### 2.1 子项目定位

Phase Q 选择实现 `社区经验帖结构化增强`，原因如下：

1. `FR-COMMUNITY-002` 已明确要求支持经验帖结构化字段，并在首页推荐区中获得更高展示权重。
2. 当前社区帖子只有标题、正文和标签，经验内容缺少统一摘要结构，用户需要全文扫描才能判断是否值得阅读。
3. Phase F 发现页排序已经存在统一热度模型，适合作为经验帖权重提升的接入点，不需要新增单独推荐系统。

### 2.2 首期设计原则

本期按以下原则收口：

- 经验帖仍然属于普通社区帖子，不新增独立帖子类型页面。
- 采用一套通用结构化模板，不按 `CAREER / EXAM / ABROAD` 拆成多套表单。
- 继续复用现有 `tag` 作为方向标签，不新增第二套方向枚举。
- 结构化字段全部为可选，未填写时帖子行为与现状保持一致。
- 推荐加权只影响首页推荐区与发现页，不影响搜索结果排序。
- 不引入经验帖审核流、运营置顶流或复杂质量评分体系。

## 3. 范围定义

### 3.1 本期范围

本期包含：

- 社区帖子新增经验帖标记字段
- 社区帖子新增经验结构化字段
- 发帖页支持选择“这是一篇经验帖”并填写结构化摘要
- 社区列表卡片显示经验帖标识与结构化摘要
- 社区详情页显示完整经验结构化区块
- 发现页 / 首页推荐复用经验帖加权后的排序结果

### 3.2 明确不做

以下能力不属于本期范围：

- 独立经验帖频道或独立经验帖路由
- 多模板分流表单
- 基于经验帖的搜索筛选和单独排序
- 经验帖可信度评分、内容质检或机器摘要
- 管理端经验帖专项审核功能
- 经验帖导出、合集页或专题运营页

## 4. 数据模型设计

### 4.1 数据落点

经验帖字段直接挂在现有 `t_community_post` 表上，不新增扩展表。

原因如下：

1. 当前社区帖子体量与能力边界较小，单表扩展足以覆盖首期诉求。
2. 经验帖字段数量有限，且只在社区帖子上下文中使用，没有必要过早拆表。
3. 这样可以让列表、详情、发现排序继续复用既有查询链路。

### 4.2 新增字段

建议新增以下字段：

- `is_experience_post`
  - 类型：`TINYINT`
  - 含义：是否开启经验帖结构化增强
  - 默认值：`0`
- `experience_target_label`
  - 类型：`VARCHAR(120)`
  - 含义：目标对象或目标事项
  - 示例：`后端开发暑期实习`、`计算机学硕`、`英国授课型硕士`
- `experience_outcome_label`
  - 类型：`VARCHAR(120)`
  - 含义：结果摘要
  - 示例：`拿到 2 个 offer`、`一战上岸`、`已提交 5 所申请`
- `experience_timeline_summary`
  - 类型：`VARCHAR(255)`
  - 含义：时间线摘要
  - 示例：`3 月定方向，4-6 月集中准备，7 月开始投递`
- `experience_action_summary`
  - 类型：`VARCHAR(500)`
  - 含义：关键动作 / 建议摘要
  - 示例：`先补项目，再集中刷八股，投递后统一复盘面试问题`

### 4.3 字段语义约束

- 当 `is_experience_post = 0` 时，其余结构化字段允许为空。
- 当 `is_experience_post = 1` 时，四个结构化字段仍允许为空，但前端会引导尽量填写，后端会对非空字段做 `trim + max length` 约束。
- 方向归类继续使用帖子已有 `tag`，不重复存储“就业 / 考研 / 留学”。

## 5. 交互与页面设计

### 5.1 发帖页

发帖页新增一个“经验帖增强”区块，交互如下：

1. 默认关闭经验帖模式。
2. 用户开启后，出现四个结构化输入项：
   - 目标
   - 结果
   - 时间线
   - 关键建议
3. 用户仍需填写标题、正文、方向标签，结构化区块只是增强信息。

首期不做分步表单，也不做自动生成文案。

### 5.2 社区列表

社区列表中的经验帖卡片新增：

- `经验帖` 徽标
- 一行或两行结构化摘要

摘要优先展示：

1. `目标`
2. `结果`
3. `时间线`

若字段为空，则退化为正文摘要，不额外制造空白占位。

### 5.3 社区详情

详情页在正文上方新增“经验摘要”区块，仅当帖子为经验帖时展示。
区块包含：

- 目标
- 结果
- 时间线
- 关键建议

这样用户可以先快速判断帖子的适用场景，再阅读正文细节。

## 6. 接口与 DTO 设计

### 6.1 创建帖子请求

`POST /api/community/posts` 在现有字段基础上增加：

```json
{
  "tag": "CAREER",
  "title": "秋招后端岗复盘",
  "content": "正文内容",
  "experiencePost": true,
  "experienceTargetLabel": "后端开发秋招",
  "experienceOutcomeLabel": "拿到 2 个 offer",
  "experienceTimelineSummary": "4 月准备项目，6 月投递，8 月集中面试",
  "experienceActionSummary": "先补项目和简历，再按面经分类复盘"
}
```

### 6.2 列表与详情响应

社区列表与详情返回统一增加 `experience` 结构：

```json
{
  "experience": {
    "enabled": true,
    "targetLabel": "后端开发秋招",
    "outcomeLabel": "拿到 2 个 offer",
    "timelineSummary": "4 月准备项目，6 月投递，8 月集中面试",
    "actionSummary": "先补项目和简历，再按面经分类复盘"
  }
}
```

当帖子不是经验帖时：

- `experience.enabled = false`
- 其余字段返回 `null`

这样前端只需要维护一套读取逻辑。

## 7. 推荐与排序设计

### 7.1 加权范围

经验帖加权只作用于：

- `GET /api/discover`
- 首页 `/api/home/summary` 中的 `discoverPreview`

不作用于：

- 社区列表默认排序
- 统一搜索排序
- 社区热榜 `GET /api/community/hot`

### 7.2 加权方式

在现有社区帖子热度分基础上，新增一个稳定的经验帖奖励分：

- `experiencePostBonus = 4.0`

排序公式调整为：

`postHotScore = like*3 + comment*4 + favorite*5 + verifiedAuthorBonus + experiencePostBonus + freshnessBonus`

这样做的原因：

1. 奖励分足以影响接近分值的帖子排序。
2. 奖励分不会压倒高互动资源或岗位内容，避免推荐结果失真。
3. 规则确定、可测试、实现成本低。

### 7.3 文案表现

发现页中的经验帖仍然以 `POST` 类型展示，但建议：

- `secondaryMeta` 追加 `经验帖`
- `hotLabel` 在帖子为经验帖时优先显示为：
  - `本周经验推荐`
  - `长期经验沉淀`

这样首页推荐卡片可以让用户更容易识别这类内容为何被前置展示。

## 8. 校验与错误处理

- `title`、`tag`、`content` 继续沿用当前必填规则。
- 结构化字段均允许为空，但若传入则需要：
  - 去除首尾空格
  - 满足最大长度限制
- 若经验帖标记关闭，后端应忽略前端误传的空白结构化内容并统一标准化存储。

## 9. 测试策略

### 9.1 后端

需要覆盖：

- 创建经验帖时结构化字段正确持久化
- 普通帖子仍可正常创建且结构化字段为空
- 列表 / 详情接口返回 `experience` 结构
- 发现页中经验帖获得加权后排序提升

### 9.2 前端

需要覆盖：

- 发帖页切换经验帖模式
- 提交经验帖时 payload 正确
- 列表页经验帖徽标与摘要展示
- 详情页经验摘要区块展示

## 10. 兼容性与发布说明

- 该变更对旧帖子兼容，历史数据默认 `is_experience_post = 0`。
- 旧前端若未发送经验字段，后端仍可正常创建帖子。
- 新前端上线后，经验帖能力从创建、展示到推荐加权形成闭环。
