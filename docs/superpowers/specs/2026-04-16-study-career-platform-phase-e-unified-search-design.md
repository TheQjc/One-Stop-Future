# 校园一站式信息平台 Phase E 统一搜索模块设计说明

> **Validation note:** This design was implemented and validated on 2026-04-16. Execution record: `docs/superpowers/plans/2026-04-16-study-career-platform-phase-e-unified-search-implementation.md`. Documented verification covered `SearchControllerTests`, `SearchView.spec.js`, `HomeView.spec.js`, `NavBar.spec.js`, full backend/frontend regression, and local smoke.
## 1. 文档目标

本文档用于确认在 Phase A 基础能力、Phase B 社区、Phase C 岗位聚合、Phase D 资料库首期闭环已经完成之后，下一阶段子项目 `统一搜索首期` 的设计边界。

本期目标不是一次性做完正式需求中的全部搜索能力，也不是立即接入 Elasticsearch，而是在现有 Spring Boot + Vue 基础上，优先交付一个：

- 可从首页和导航进入
- 可按关键词搜索三类公开内容
- 可切换结果类型
- 可切换基础排序
- 可稳定承接后续 Elasticsearch 替换

的最小统一搜索闭环。

## 2. 当前结论

### 2.1 子项目定位

Phase E 选择 `统一搜索首期` 作为下一子项目，原因如下：

1. 社区、岗位、资料三类公开内容已经具备可检索的数据源，统一搜索现在才有稳定输入面。
2. 统一搜索比推荐、热榜、运营看板更基础，能够先把平台从“按模块找内容”推进到“按问题找内容”。
3. 统一搜索首期不必强依赖 Elasticsearch，适合先以数据库模糊查询形成产品闭环，再为后续搜索引擎替换保留统一接口。

### 2.2 首期交付原则

统一搜索首期按以下原则收口：

- 先做公开内容搜索，不引入个人化结果。
- 游客可用，但只能搜索公开内容。
- 结果页使用独立 `/search` 路由，不在首页直接内嵌结果流。
- 首页顶部增加搜索框，但搜索结果统一落到独立搜索页。
- 第一版支持 `关键词 + 类型切换 + 基础排序`，不做高亮和复杂筛选。
- 后端采用统一聚合接口，不把聚合逻辑堆到前端。
- Elasticsearch 本期不接入正式链路，但接口和服务边界要为后续替换保留稳定协议。

## 3. 范围定义

### 3.1 本期范围

本期统一搜索模块包含：

- 首页顶部搜索框
- 导航栏搜索入口
- 独立搜索结果页 `/search`
- 公共搜索接口 `GET /api/search`
- 三类公开内容聚合搜索：
  - 社区帖子
  - 岗位信息
  - 资料资源
- 结果类型切换：
  - `ALL`
  - `POST`
  - `JOB`
  - `RESOURCE`
- 基础排序切换：
  - `RELEVANCE`
  - `LATEST`
- 统一结果统计与统一结果卡片模型

### 3.2 明确不做

以下能力不属于本期范围：

- Elasticsearch 正式接入
- 关键词高亮
- 搜索建议 / 联想
- 搜索历史
- 热门搜索词
- 高级筛选器
- 分页之外的复杂无限滚动交互
- 个性化推荐结果
- 登录后额外注入私有内容
- 多字段权重配置后台

## 4. 角色与权限

| 角色 | 可使用统一搜索 | 可搜索范围 |
| --- | --- | --- |
| 游客 | 是 | 仅公开社区帖子、已发布岗位、已发布资料 |
| 普通用户 | 是 | 与游客一致 |
| 已认证用户 | 是 | 与游客一致 |
| 管理员 | 是 | 搜索页仍默认只搜公开内容 |

补充规则：

- 本期统一搜索不暴露后台内容、待审核内容、草稿内容和下架内容。
- 管理员在搜索页也不额外看到管理态内容，后台检索不是本期范围。
- 搜索入口对游客开放，不设置登录门槛。

## 5. 信息架构

### 5.1 入口与路由

统一搜索首期页面结构如下：

- `/`
  首页顶部新增搜索框，提交后跳转 `/search`
- `/search`
  统一搜索结果页

首页和导航的行为规则如下：

- 首页顶部搜索框始终可见
- 顶部导航新增固定 `搜索` 入口
- 首页搜索提交默认跳转：
  - `/search?q=<keyword>&type=ALL&sort=RELEVANCE`
- 导航点击 `搜索` 时：
  - 若已有当前搜索上下文，可保留当前参数
  - 若无参数，则进入空状态搜索页 `/search`

### 5.2 搜索页结构

`/search` 页面分为三层：

1. 顶部搜索区
   - 主搜索框
   - 搜索按钮
   - 搜索说明文案
2. 结果控制区
   - 类型切换：`全部 / 社区 / 岗位 / 资料`
   - 排序切换：`相关度 / 最新`
   - 命中总数摘要
3. 结果列表区
   - 统一卡片流
   - 明确类型标签
   - 统一跳转到原始详情页

### 5.3 与现有模块的衔接方式

- 社区结果点击后跳转 `/community/:id`
- 岗位结果点击后跳转 `/jobs/:id`
- 资料结果点击后跳转 `/resources/:id`
- 首页与导航只作为搜索入口，不承载聚合结果渲染
- 搜索页仅消费统一搜索接口，不直接并行拼接三个现有列表接口

## 6. 搜索模型设计

### 6.1 固定枚举

搜索类型首期固定为：

- `ALL`
- `POST`
- `JOB`
- `RESOURCE`

搜索排序首期固定为：

- `RELEVANCE`
- `LATEST`

### 6.2 统一结果对象

建议新增统一搜索结果对象 `SearchResultItem`，至少包含：

- `id`
- `type`
- `title`
- `summary`
- `metaPrimary`
- `metaSecondary`
- `path`
- `publishedAt`

字段语义建议如下：

- `type`
  表示结果来源类型：帖子 / 岗位 / 资料
- `title`
  对应原内容主标题
- `summary`
  对应原内容摘要或正文预览
- `metaPrimary`
  对应结果最主要的辅信息
  - 帖子：作者昵称
  - 岗位：公司名
  - 资料：上传者昵称
- `metaSecondary`
  对应结果次级分类信息
  - 帖子：标签
  - 岗位：城市 / 岗位类型组合
  - 资料：资料分类
- `path`
  前端直接跳转目标
- `publishedAt`
  用于统一排序和时间显示

### 6.3 统计对象

统一搜索响应应返回固定统计对象：

- `all`
- `post`
- `job`
- `resource`

用于前端渲染类型切换标签与结果数量，不要求前端自行二次统计。

## 7. 接口边界

### 7.1 公共接口

统一搜索接口建议如下：

- `GET /api/search`

支持参数：

- `q`
  - 必填关键词
  - 去首尾空格
  - 为空时返回业务码 `400`
- `type`
  - 可选
  - 默认 `ALL`
- `sort`
  - 可选
  - 默认 `RELEVANCE`

建议响应结构：

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "query": "resume",
    "type": "ALL",
    "sort": "RELEVANCE",
    "totals": {
      "all": 6,
      "post": 2,
      "job": 1,
      "resource": 3
    },
    "results": [
      {
        "id": 11,
        "type": "RESOURCE",
        "title": "2026 Resume Template Pack",
        "summary": "A concise resume starter pack...",
        "metaPrimary": "VerifiedUser",
        "metaSecondary": "RESUME_TEMPLATE",
        "path": "/resources/11",
        "publishedAt": "2026-04-16T12:30:00"
      }
    ]
  }
}
```

### 7.2 后端职责拆分

后端建议新增以下边界：

- `SearchController`
  - 负责参数接收
  - 负责基本校验
  - 返回统一结果
- `SearchService`
  - 负责聚合三类公开内容
  - 负责统一映射与排序

为避免破坏现有领域边界，聚合搜索的只读查询建议通过各领域 service 暴露搜索方法：

- `CommunityService.searchPublishedPosts(...)`
- `JobService.searchPublishedJobs(...)`
- `ResourceService.searchPublishedResources(...)`

统一搜索服务只组合这些公开结果，不直接承担各领域业务规则判定。

## 8. 查询与排序规则

### 8.1 查询规则

本期采用数据库模糊查询实现，规则如下：

- 社区：
  - 搜索 `title`
  - 搜索 `content`
  - 仅返回 `PUBLISHED`
- 岗位：
  - 搜索 `title`
  - 搜索 `companyName`
  - 搜索 `summary`
  - 搜索 `content`
  - 仅返回 `PUBLISHED`
- 资料：
  - 搜索 `title`
  - 搜索 `summary`
  - 搜索 `description`
  - 仅返回 `PUBLISHED`

### 8.2 相关度排序

`RELEVANCE` 排序建议使用简单分层，不在本期引入完整打分引擎：

1. 标题命中最高
2. 摘要类字段命中次之
3. 正文字段命中再次之
4. 同层内按时间倒序

这样可以在不引入 Elasticsearch 的前提下，先保证首期结果顺序具备基本可解释性。

### 8.3 最新排序

`LATEST` 排序统一按时间倒序：

- 帖子使用 `createdAt`
- 岗位使用 `publishedAt`
- 资料使用 `publishedAt`

若某类内容缺少公开时间字段，不纳入首期公开搜索结果。

## 9. 前端交互设计

### 9.1 首页顶部搜索框

首页顶部新增搜索框，行为如下：

- 支持输入关键词后回车提交
- 支持点击搜索按钮提交
- 提交后跳转搜索页
- 不在首页直接展开结果列表

### 9.2 搜索页行为

搜索页行为规则如下：

- 从 URL 读取 `q / type / sort`
- 参数变化即重新请求
- 类型切换和排序切换都要同步回 URL
- 刷新页面后结果保持一致
- 浏览器前进 / 后退可回到上一次搜索状态

### 9.3 页面状态

搜索页至少覆盖以下状态：

- 空状态
  - 没有关键词时显示引导文案
- 加载态
  - 仅阻塞结果区，不阻塞搜索框继续输入
- 无结果态
  - 明确提示“没有匹配结果”
- 错误态
  - 展示统一错误文案与重试按钮

### 9.4 移动端约束

- 顶部搜索区在移动端允许纵向堆叠
- 类型切换按钮允许换行
- 排序切换保持单行或换行，但不引入复杂侧边栏
- 结果列表保持单列卡片流

## 10. 错误处理

至少覆盖以下规则：

- `q` 为空时返回业务码 `400`
- `type` 非法时返回业务码 `400`
- `sort` 非法时返回业务码 `400`
- 聚合查询异常时返回统一错误结构
- 统一搜索继续沿用当前项目的统一响应格式：

```json
{
  "code": 200,
  "message": "success",
  "data": {}
}
```

以及统一业务错误结构：

```json
{
  "code": 400,
  "message": "invalid search type",
  "data": null
}
```

本期不做“部分成功、部分失败”的混合搜索返回协议。

## 11. 测试与验收

### 11.1 后端测试重点

- 空关键词返回 `400`
- 非法 `type` 返回 `400`
- 非法 `sort` 返回 `400`
- `ALL` 查询可聚合三类公开内容
- `POST / JOB / RESOURCE` 单类型查询正确
- `RELEVANCE` 排序优先标题命中
- `LATEST` 排序按统一时间字段倒序
- 非公开内容不会进入搜索结果

### 11.2 前端测试重点

- 首页搜索框提交后跳转 `/search`
- 搜索页根据 URL 参数发起查询
- 类型切换会更新 URL 并刷新结果
- 排序切换会更新 URL 并刷新结果
- 空状态、无结果、错误状态能正确渲染
- 结果卡片能跳转原始详情页

### 11.3 本期验收标准

当满足以下条件时，可判定统一搜索首期达到可实施状态：

1. 游客可从首页和导航进入统一搜索
2. 输入关键词后可在单页结果中看到社区、岗位、资料三类公开内容
3. 可在 `全部 / 社区 / 岗位 / 资料` 之间切换
4. 可在 `相关度 / 最新` 之间切换
5. 搜索结果点击后能进入原内容详情页
6. 搜索页 URL 可分享、可刷新、可回退
7. 接口和页面结构为后续 Elasticsearch 替换保留稳定边界

## 12. 实施顺序建议

建议按以下顺序实施：

1. 先新增搜索枚举、DTO 和聚合接口边界
2. 再为社区、岗位、资料补充领域内公开搜索方法
3. 再实现统一搜索 service 的聚合与排序
4. 再接前端 `/search` 路由和 API
5. 再接首页顶部搜索框与导航入口
6. 最后补测试、README 和本地 smoke
