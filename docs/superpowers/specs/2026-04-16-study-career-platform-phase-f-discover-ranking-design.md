# 校园一站式信息平台 Phase F 热榜与推荐首期设计说明

> **Validation note:** This design was implemented and validated on 2026-04-16. Execution record: `docs/superpowers/plans/2026-04-16-study-career-platform-phase-f-discover-ranking-implementation.md`. Documented verification covered `DiscoverControllerTests`, `HomeControllerTests`, `HomeServiceTests`, `NavBar.spec.js`, `DiscoverView.spec.js`, `HomeView.spec.js`, full backend/frontend regression, and local smoke.
## 1. 文档目标

本文档用于确认在 Phase E 统一搜索首期完成之后，下一个子项目 `热榜与推荐首期` 的设计边界。
本期目标不是一次性做完整推荐系统、画像系统或运营看板，而是在现有 Spring Boot + Vue 基础上，优先交付一个：

- 可从首页和导航进入
- 可稳定浏览跨域热门内容
- 可按内容域和时间范围切换
- 可复用现有社区 / 岗位 / 资料数据
- 可为后续个性化推荐、缓存榜单和运营配置扩展留口

的最小发现闭环。

## 2. 当前结论

### 2.1 子项目定位

Phase F 选择 `热榜与推荐首期` 作为下一子项目，原因如下：

1. 社区、岗位、资料三类公开内容已经具备可聚合的发布数据和基础行为数据，适合进入“发现”视图。
2. 在统一搜索完成后，平台已经具备“主动找内容”能力，下一步更合理的是补上“被动发现内容”能力。
3. 热榜首期可以不依赖画像、埋点平台或调度任务，适合先用轻量实现形成产品闭环。

### 2.2 首期交付原则

热榜与推荐首期按以下原则收口：

- 先做非个性化发现，不做千人千面。
- 游客可浏览，登录状态不改变榜单排序结果。
- 首页只做轻量推荐区，不在首页展开完整榜单交互。
- 使用独立 `/discover` 路由承载完整榜单浏览。
- 榜单时间范围首期固定为 `本周榜 + 总榜`，不提前引入日榜。
- 榜单数据直接来自现有公开内容和已有行为数据，不新增运营手工编排后台。
- 首页推荐区和 `/discover` 页面共用同一套发现项结构与排序逻辑。
- 本期不引入 Redis 榜单缓存、离线预计算任务或推荐解释系统。

## 3. 范围定义

### 3.1 本期范围

本期热榜与推荐模块包含：

- 顶部导航新增 `发现` 入口
- 首页新增轻量 `热门推荐` 区块
- 独立发现页 `/discover`
- 公共榜单接口 `GET /api/discover`
- 首页聚合接口 `/api/home/summary` 扩展推荐预览字段
- 跨域榜单聚合，覆盖：
  - 社区帖子
  - 岗位信息
  - 资料资源
- 榜单内容域切换：
  - `ALL`
  - `POST`
  - `JOB`
  - `RESOURCE`
- 榜单时间范围切换：
  - `WEEK`
  - `ALL`
- 首页预览与发现页复用统一发现项卡片模型

### 3.2 明确不做

以下能力不属于本期范围：

- 个性化推荐
- 用户画像或偏好建模
- 日榜
- 后台运营手工置顶或手工配置推荐位
- 点击率、曝光率等完整推荐埋点体系
- Redis / 定时任务 / 榜单快照表
- “猜你喜欢”“相关推荐”等详情页内联推荐
- 热门搜索词与搜索榜联动
- 复杂推荐理由解释系统
- 完整看板化运营分析页面

## 4. 角色与权限

| 角色 | 可使用首页热门推荐 | 可访问 `/discover` | 可见内容范围 |
| --- | --- | --- | --- |
| 游客 | 是 | 是 | 仅公开社区帖子、已发布岗位、已发布资料 |
| 普通用户 | 是 | 是 | 与游客一致 |
| 已认证用户 | 是 | 是 | 与游客一致 |
| 管理员 | 是 | 是 | 发现页仍默认只展示公开内容 |

补充规则：

- 本期发现页不暴露草稿、待审核、已下架或后台专用内容。
- 管理员在发现页不额外看到管理态榜单，后台榜单不属于本期范围。
- 游客无需登录即可进入 `/discover`，不设置登录门槛。

## 5. 信息架构

### 5.1 入口与路由

热榜与推荐首期页面结构如下：

- `/`
  首页新增 `热门推荐` 区块，默认展示本周榜前若干条内容
- `/discover`
  独立发现页，承载跨域热榜浏览与切换

入口行为规则如下：

- 顶部导航固定新增 `发现` 入口
- 首页推荐区展示 4 到 6 条内容，首期建议固定返回 4 条
- 点击首页推荐项后，直接跳转到原始详情页
- 点击首页 `进入发现页` 后，跳转到默认榜单：
  - `/discover?tab=ALL&period=WEEK`
- 导航点击 `发现` 时：
  - 若当前无上下文，默认进入 `/discover?tab=ALL&period=WEEK`
  - 若已在发现页内部切换过参数，则由路由自身维持当前查询参数

### 5.2 发现页结构

`/discover` 页面分为四层：

1. 顶部说明区
   - 页面标题
   - 说明文案
   - 当前榜单口径提示
2. 时间范围切换区
   - `本周榜`
   - `总榜`
3. 内容域切换区
   - `全部`
   - `社区`
   - `岗位`
   - `资料`
4. 榜单结果区
   - 热榜卡片列表
   - 加载态
   - 空态
   - 错误态

页面行为规则如下：

- 默认进入 `ALL + WEEK`
- `tab` 与 `period` 变化都要同步回 URL query
- 刷新页面后状态保持一致
- 浏览器前进 / 后退可恢复榜单上下文
- 榜单项点击后跳转到原始详情页，不在发现页内二次展开详情

### 5.3 与首页的衔接方式

首页 `热门推荐` 区块使用和 `/discover` 相同的发现项结构，但只展示少量预览内容，不承担完整切换交互。

首页推荐区的职责如下：

- 告诉用户平台已经具备“发现”能力
- 快速暴露当前值得看的公开内容
- 引导用户进入 `/discover` 完成进一步浏览

首页推荐区不承担以下职责：

- 不做完整榜单分页
- 不做内容域切换
- 不做时间范围切换
- 不做“更多推荐算法”解释

## 6. 接口与数据模型

### 6.1 公共榜单接口

发现页接口定义如下：

- `GET /api/discover`

支持参数：

- `tab`
  - 可选
  - 默认 `ALL`
  - 允许值：`ALL / POST / JOB / RESOURCE`
- `period`
  - 可选
  - 默认 `WEEK`
  - 允许值：`WEEK / ALL`
- `limit`
  - 可选
  - 默认 `20`
  - 建议限制在 `1 - 50`

建议响应结构：

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "tab": "ALL",
    "period": "WEEK",
    "total": 8,
    "items": [
      {
        "id": 21,
        "type": "POST",
        "title": "应届生春招复盘",
        "summary": "整理了投递节奏、面试安排和复盘建议。",
        "primaryMeta": "认证用户",
        "secondaryMeta": "CAREER",
        "path": "/community/21",
        "publishedAt": "2026-04-16T09:30:00",
        "hotScore": 18.5,
        "hotLabel": "本周热议"
      }
    ]
  }
}
```

### 6.2 首页聚合接口扩展

首页继续使用现有：

- `GET /api/home/summary`

但在原有聚合结构上新增：

- `discoverPreview`

建议该字段包含：

- `period`
  - 首期固定返回 `WEEK`
- `items`
  - 推荐预览列表

`discoverPreview.items` 与 `/api/discover` 的 `items` 复用同一发现项结构，避免前端为首页和发现页维护两套数据模型。

### 6.3 统一发现项结构

建议新增统一发现项对象 `DiscoverItem`，至少包含：

- `id`
- `type`
- `title`
- `summary`
- `primaryMeta`
- `secondaryMeta`
- `path`
- `publishedAt`
- `hotScore`
- `hotLabel`

字段语义建议如下：

- `id`
  对应原始内容主键
- `type`
  表示结果来源类型：帖子 / 岗位 / 资料
- `title`
  对应原始内容标题
- `summary`
  对应摘要或正文预览
- `primaryMeta`
  对应最主要的辅助信息
  - 帖子：作者昵称或认证状态文案
  - 岗位：公司名
  - 资料：上传者昵称
- `secondaryMeta`
  对应次级信息
  - 帖子：标签
  - 岗位：城市 / 岗位类型 / 来源平台组合
  - 资料：分类
- `path`
  前端直接跳转目标
- `publishedAt`
  用于统一排序和时间展示
- `hotScore`
  用于后端排序、调试和测试校验，不强制直接展示给用户
- `hotLabel`
  用于向用户表达榜单语义，例如 `本周热议`、`长期热门`、`高频下载`

### 6.4 页面数据流

首页推荐预览的数据流建议如下：

1. 前端进入首页并请求 `/api/home/summary`
2. `HomeService` 聚合身份信息、通知、入口卡片等既有首页数据
3. `HomeService` 调用 `DiscoverService` 获取固定条数的 `WEEK` 推荐预览
4. `DiscoverService` 从社区、岗位、资料三个域收集公开候选内容
5. `DiscoverService` 统一算分、排序、裁剪
6. 首页返回包含 `discoverPreview` 的聚合响应

发现页的数据流建议如下：

1. 前端从 URL 读取 `tab / period / limit`
2. 前端请求 `/api/discover`
3. `DiscoverController` 完成参数校验
4. `DiscoverService` 拉取候选数据、过滤时间范围、计算热度并统一排序
5. 后端返回统一发现项列表
6. 前端按当前 query 渲染榜单卡片、空态或错误态

## 7. 排行模型与数据来源

### 7.1 时间范围口径

本期时间范围定义如下：

- `WEEK`
  - 表示最近 7 天滚动窗口
  - 仅纳入 `publishedAt >= now - 7 days` 的公开内容
- `ALL`
  - 表示全部历史公开内容

这里的 `WEEK` 明确采用滚动 7 天窗口，而不是自然周，避免周一整榜跳变。
由于当前没有“最近 7 天行为增量流水”，本期 `WEEK` 的语义明确为：

- 最近 7 天发布的内容
- 按当前累计热度排序

而不是：

- 所有历史内容
- 按最近 7 天新增点赞 / 收藏 / 下载量排序

### 7.2 跨域聚合原则

发现页的榜单来源为三个公开域：

- 社区帖子
- 岗位信息
- 资料资源

聚合原则如下：

- 仅纳入已发布内容
- 每个域先按各自公式算出热度分
- 再统一按 `hotScore DESC, publishedAt DESC, id DESC` 排序
- `ALL` 视图直接合并三域结果后统一排序
- 单域视图仅返回所选域结果

### 7.3 社区热度公式

社区已有可直接复用的行为字段：

- `likeCount`
- `commentCount`
- `favoriteCount`
- `createdAt`

首期建议社区热度公式为：

- `rawHeat = likeCount * 3 + commentCount * 4 + favoriteCount * 5 + verifiedAuthorBonus`
- `verifiedAuthorBonus = 2`，仅当作者为已认证用户时生效，否则为 `0`
- `publishedAt = createdAt`
- `hotScore = rawHeat + freshnessBonus`

其中：

- `freshnessBonus = max(0, 14 - ageDays)`
- `ageDays` 为内容发布时间距当前时间的自然天数差

补充说明：

- 当前社区数据结构尚不能稳定识别“经验帖”这一独立帖子类型，因此首期不新增帖子类型字段。
- 用户已经确认的“小幅可信内容加权”在当前实现里由 `认证作者加权` 承担。
- 后续若社区帖子新增经验帖结构化字段，可在不破坏接口的前提下把可信权重从“认证作者代理”升级为“经验帖 + 认证作者组合权重”。

### 7.4 岗位热度公式

岗位当前没有独立热度计数字段，但已有：

- `publishedAt`
- `t_user_favorite` 中的岗位收藏关系

首期岗位热度公式建议为：

- `favoriteCount = 岗位在 t_user_favorite 中的收藏次数`
- `rawHeat = favoriteCount * 5`
- `hotScore = rawHeat + freshnessBonus`

其中：

- `freshnessBonus = max(0, 14 - ageDays)`
- 若岗位收藏次数为 `0`，仍可凭发布时间参与排序

这样可以满足两个目标：

- 首期不新增岗位统计表
- 在行为数据不足时，以“发布时间衰减”为主保证榜单可用

### 7.5 资料热度公式

资料域已有：

- `downloadCount`
- `favoriteCount`
- `publishedAt`

首期资料热度公式建议为：

- `rawHeat = downloadCount * 2 + favoriteCount * 4`
- `hotScore = rawHeat + freshnessBonus`

其中：

- `freshnessBonus = max(0, 14 - ageDays)`

### 7.6 热度标签映射

`hotLabel` 首期不需要复杂机器生成，可按类型和时间口径映射：

- 社区 + WEEK：`本周热议`
- 社区 + ALL：`长期热议`
- 岗位 + WEEK：`本周关注`
- 岗位 + ALL：`持续关注`
- 资料 + WEEK：`本周高频下载`
- 资料 + ALL：`长期热门资料`

`ALL` 视图不改变单条内容自身的 `hotLabel`，仍按其来源域生成。

## 8. 后端边界设计

### 8.1 控制器与服务职责

后端建议新增以下边界：

- `DiscoverController`
  - 负责参数接收
  - 负责基本参数校验
  - 返回统一发现页结果
- `DiscoverService`
  - 负责聚合三域公开内容
  - 负责热度计算
  - 负责时间范围过滤
  - 负责统一排序和裁剪

首页聚合仍由 `HomeService` 负责，但其内部应通过 `DiscoverService` 读取首页推荐预览，而不是自行复制一套榜单逻辑。

### 8.2 与现有域服务的协作方式

为避免破坏现有模块边界，发现聚合建议通过现有域服务暴露只读候选数据方法，而不是让 `DiscoverService` 直接承担领域规则判断。

建议由以下服务补充只读聚合能力：

- `CommunityService`
  - 暴露公开帖子发现候选项
- `JobService`
  - 暴露公开岗位发现候选项
  - 聚合岗位收藏次数
- `ResourceService`
  - 暴露公开资料发现候选项

`DiscoverService` 只做：

- 聚合
- 统一映射
- 算分
- 排序

不直接承接各域写操作逻辑。

### 8.3 首页降级策略

首页推荐预览属于聚合块，不应影响首页主链路可用性。

建议策略如下：

- `HomeService` 调用 `DiscoverService` 获取预览数据
- 若推荐预览计算失败：
  - 记录错误日志
  - `discoverPreview.items` 返回空列表
  - `/api/home/summary` 主响应仍然成功
- `/api/discover` 自身失败时：
  - 显式返回错误
  - 不返回伪造榜单数据

这样可以保证首页作为核心入口的稳定性，同时不掩盖发现页真实错误。

## 9. 前端交互设计

### 9.1 导航入口

顶部导航新增固定 `发现` 入口，位置建议与现有 `搜索` 并列。

设计原则如下：

- `发现` 是内容浏览入口
- `搜索` 是问题检索入口
- 两者并列，而不是互相替代

### 9.2 首页热门推荐区

首页新增轻量 `热门推荐` 区块，建议位于现有首页聚合信息区的中上段，避免过深埋藏。

该区块建议包含：

- 区块标题
- 一句简短说明文案
- 4 条推荐项
- `进入发现页` 按钮

交互规则如下：

- 不提供 tab 切换
- 不提供 period 切换
- 每条卡片可直接跳转到原始详情页
- 区块为空时展示自然空态文案，不隐藏整块结构

### 9.3 发现页状态设计

发现页至少覆盖以下状态：

- 默认态
  - 展示 `ALL + WEEK`
- 加载态
  - 仅阻塞榜单区，不阻塞页面框架
- 空态
  - 提示“当前口径下还没有公开内容进入榜单”
- 错误态
  - 展示统一错误文案与重试入口

页面空态文案不应暴露“算法未就绪”“推荐系统异常”等开发语义，而应保持产品语义完整。

### 9.4 组件边界建议

前端建议新增：

- `DiscoverView.vue`
  - 发现页容器
- `DiscoverItemCard.vue`
  - 统一发现项卡片
- `api/discover.js`
  - 发现页接口访问层

首页推荐区与 `/discover` 复用 `DiscoverItemCard`，避免首页和发现页为同一数据维护两种视图实现。

## 10. 错误处理与降级

本期至少覆盖以下异常场景：

- `tab` 非法
  - 返回业务码 `400`
- `period` 非法
  - 返回业务码 `400`
- `limit` 非法或超范围
  - 返回业务码 `400`
- 某一内容域当前无数据
  - 不报错
  - `ALL` 视图自然缺席
  - 单域页返回空列表
- 首页推荐预览聚合异常
  - 首页接口不失败
  - 预览区空数组降级
- 发现页接口聚合异常
  - 发现页显式错误态

统一错误响应结构继续保持：

- `{code, message, data}`

## 11. 测试与验收

### 11.1 后端测试重点

- `WEEK / ALL` 时间口径过滤正确
- `ALL / POST / JOB / RESOURCE` 内容域过滤正确
- 社区、岗位、资料三类内容热度公式正确
- 相同分数下按发布时间倒序
- 首页推荐预览复用与发现页一致的发现项结构
- 首页推荐预览聚合失败时能正确降级为空列表
- 非法参数返回统一业务错误

### 11.2 前端测试重点

- 导航新增 `发现` 入口
- `/discover` 默认 query 为 `tab=ALL&period=WEEK`
- tab 切换同步 URL
- period 切换同步 URL
- 刷新后榜单状态保持
- 首页热门推荐区正确渲染
- 首页 `进入发现页` 跳转正确
- 空态与错误态文案正确展示

### 11.3 本期验收标准

满足以下条件时，可判定 Phase F 进入可实施状态：

- 游客可访问首页热门推荐区和 `/discover`
- 导航存在稳定 `发现` 入口
- `/discover` 默认进入 `全部 + 本周榜`
- 榜单可切换 `全部 / 社区 / 岗位 / 资料`
- 榜单可切换 `本周榜 / 总榜`
- 首页推荐区与发现页使用同一套发现项结构
- 社区、岗位、资料三类内容在有数据时都可进入榜单
- 首页推荐预览异常不会影响首页其他模块渲染

## 12. 后续扩展位

本设计为后续能力预留以下稳定扩展位：

- 日榜
- 个性化推荐
- 经验帖结构化可信权重
- Redis 或榜单快照缓存
- 运营手工置顶与推荐位配置
- 详情页相关推荐
- 推荐埋点与效果分析

这些能力都应在后续新一轮 `spec -> plan -> implementation` 流程中独立展开，不在本期提前混入实现范围。
