# 校园一站式信息平台 Phase C 岗位聚合设计说明

## 1. 文档目标

本文档用于确认 Phase B 社区模块完成后的下一阶段子项目范围，即 `岗位聚合首期`。

目标不是一次性做完正式需求中的全部 Phase C 能力，而是在现有 Spring Boot + Vue 基础上，优先交付一个可浏览、可筛选、可收藏、可外链跳转、可后台维护的最小闭环，为后续资料库和统一搜索提供稳定接入面。

## 2. 当前结论

### 2.1 子项目定位

Phase C 的第一个子项目选择 `岗位聚合`，原因如下：

1. 首页和后端聚合接口已经预留 `jobs` 入口，岗位模块可以直接接入现有壳层。
2. 岗位聚合比资料库和统一搜索更容易先形成独立闭环，且不依赖对象存储和全文检索主链路。
3. 岗位收藏可以直接复用已有通用收藏模型，为后续资料收藏和统一搜索结果聚合打底。

### 2.2 首期交付原则

岗位聚合首期按以下原则收口：

- 岗位数据由管理员手工维护，不接第三方自动同步。
- 模块只承担“聚合与导流”职责，不做站内投递、简历投递、私信咨询。
- 游客、普通用户、已认证用户都可以浏览岗位列表和详情。
- 登录用户可收藏岗位，个人中心继续复用统一收藏入口。
- 管理员负责创建、编辑、发布、下架和删除岗位。
- 统一搜索本期不落地真实能力，只保留后续接入位。

## 3. 范围定义

### 3.1 本期范围

本期岗位聚合模块包含：

- 岗位列表页
- 岗位详情页
- 关键词与条件筛选
- 岗位外链跳转
- 岗位收藏 / 取消收藏
- 个人中心岗位收藏复用
- 管理员岗位维护台
- 首页 `jobs` 入口从占位变为真实可进入模块

### 3.2 明确不做

以下能力不属于本期范围：

- 第三方岗位源自动抓取或同步
- 批量 CSV / Excel 导入
- 站内投递、简历上传、岗位报名
- 职位热榜、推荐算法、个性化排序
- 与统一搜索的真实联动
- 与资料库的跨域联动
- 岗位互动通知

## 4. 角色与权限

| 角色 | 可浏览岗位列表 / 详情 | 可筛选 | 可跳转原链接 | 可收藏岗位 | 可查看岗位收藏 | 可维护岗位 |
| --- | --- | --- | --- | --- | --- | --- |
| 游客 | 是 | 是 | 是 | 否 | 否 | 否 |
| 普通用户 | 是 | 是 | 是 | 是 | 是 | 否 |
| 已认证用户 | 是 | 是 | 是 | 是 | 是 | 否 |
| 管理员 | 是 | 是 | 是 | 是 | 是 | 是 |

补充规则：

- 岗位写操作必须依赖现有 JWT 登录态。
- 管理能力必须在后端单独做角色校验，不能只依赖前端菜单隐藏。
- 管理员在前台仍视作普通登录用户，但额外拥有岗位维护权限。

## 5. 信息架构

### 5.1 页面结构

岗位聚合首期页面结构如下：

- `/jobs`
  岗位列表页，支持关键词、城市、岗位类型、学历要求、来源平台筛选。
- `/jobs/:id`
  岗位详情页，展示整理后的摘要、补充说明、来源信息和外链跳转按钮。
- `/admin/jobs`
  管理员岗位维护页。

### 5.2 与现有 Phase A / B 的衔接方式

- 首页继续沿用现有聚合结构，`jobs` 入口不再是 `COMING_SOON` 占位。
- 个人中心不新建第二套岗位用户壳层，而是复用现有收藏能力，通过 `type=JOB` 查看岗位收藏。
- 管理后台继续沿用现有最小后台结构，在认证审核页和社区治理页之外新增岗位维护页。

## 6. 岗位模型设计

### 6.1 岗位主对象

建议新增 `JobPosting` 实体，首期只保留单表主模型，至少包含：

- `id`
- `title`
- `companyName`
- `city`
- `jobType`
- `educationRequirement`
- `sourcePlatform`
- `sourceUrl`
- `summary`
- `content`
- `deadlineAt`
- `publishedAt`
- `status`
- `createdBy`
- `updatedBy`
- `createdAt`
- `updatedAt`

字段约束建议：

- `title` 为必填，限制在 120 字以内。
- `companyName` 为必填，限制在 80 字以内。
- `city` 首期用字符串，不做城市字典表。
- `summary` 用于列表摘要，长度建议不超过 300 字。
- `content` 用于详情补充说明，可为空，但建议管理员补齐核心信息。
- `sourceUrl` 为必填，必须是可跳转外链。

### 6.2 固定枚举

岗位类型首期固定为：

- `INTERNSHIP`
- `FULL_TIME`
- `CAMPUS`

学历要求首期固定为：

- `ANY`
- `BACHELOR`
- `MASTER`
- `DOCTOR`

岗位状态定义为：

- `DRAFT`
  仅管理员可见，不进入前台列表。
- `PUBLISHED`
  前台列表和详情可见。
- `OFFLINE`
  管理员下架，前台不可见。
- `DELETED`
  软删除，作为审计状态保留。

### 6.3 收藏模型复用

不新建岗位收藏表，直接复用现有 `UserFavorite`：

- `targetType` 新增 `JOB`
- `targetId` 指向岗位主键

这样可以继续复用 `/api/users/me/favorites`，为后续资料库统一收藏保持一致结构。

## 7. 状态流转与数据流

### 7.1 岗位流转

岗位状态流转为：

1. 管理员创建岗位草稿，状态为 `DRAFT`
2. 管理员发布后，状态变为 `PUBLISHED`
3. 用户在前台浏览、筛选、查看详情并跳转外链
4. 管理员可将岗位改为 `OFFLINE` 或 `DELETED`

补充规则：

- 前台公共接口只读取 `PUBLISHED` 数据。
- `OFFLINE` 和 `DELETED` 对普通用户统一视为不存在。
- `publishedAt` 只在首次发布或重新发布时更新。

### 7.2 收藏流转

- 登录用户可收藏岗位。
- 重复收藏按幂等处理，不返回业务错误。
- 取消收藏后，个人中心岗位收藏视图同步更新。

## 8. 接口边界

### 8.1 公共 / 用户侧接口

建议岗位相关接口如下：

- `GET /api/jobs`
  获取岗位列表，支持筛选条件：
  - `keyword`
  - `city`
  - `jobType`
  - `educationRequirement`
  - `sourcePlatform`
- `GET /api/jobs/{id}`
  获取岗位详情。
- `POST /api/jobs/{id}/favorite`
  收藏岗位。
- `DELETE /api/jobs/{id}/favorite`
  取消收藏岗位。
- `GET /api/users/me/favorites?type=JOB`
  获取我的岗位收藏。

### 8.2 管理侧接口

建议后台接口如下：

- `GET /api/admin/jobs`
  获取岗位维护列表。
- `POST /api/admin/jobs`
  创建岗位草稿。
- `PUT /api/admin/jobs/{id}`
  编辑岗位。
- `POST /api/admin/jobs/{id}/publish`
  发布岗位。
- `POST /api/admin/jobs/{id}/offline`
  下架岗位。
- `POST /api/admin/jobs/{id}/delete`
  删除岗位。

后台列表至少展示：

- 岗位标题
- 公司名称
- 城市
- 岗位类型
- 学历要求
- 来源平台
- 发布时间 / 截止时间
- 状态

## 9. 前端交互设计

### 9.1 岗位列表页

列表页职责：

- 展示岗位聚合说明区
- 展示关键词与筛选条
- 展示岗位摘要卡片
- 为游客和登录用户统一提供 `查看详情`
- 为所有用户提供 `跳转原链接`
- 为登录用户提供 `收藏 / 取消收藏`

首期排序保持简单：

- 默认按 `publishedAt` 倒序
- 不做热度排序和推荐排序

### 9.2 岗位详情页

详情页职责：

- 展示完整岗位标题和公司信息
- 展示城市、岗位类型、学历要求、来源平台、发布时间、截止时间
- 展示管理员整理的摘要与补充说明
- 提供外链跳转按钮
- 登录用户可执行收藏 / 取消收藏

### 9.3 管理员岗位维护页

岗位维护页职责：

- 查看岗位维护列表
- 创建岗位草稿
- 编辑岗位内容
- 发布岗位
- 下架岗位
- 删除岗位

本期不做批量导入弹窗，不做导入模板管理，不做复杂审核流。

## 10. 异常处理与后续接入位

至少覆盖以下规则：

- 游客访问岗位收藏接口时返回 `401`
- 普通用户访问后台岗位维护接口时返回 `403`
- 普通用户访问 `DRAFT / OFFLINE / DELETED` 岗位详情时返回 `404`
- 筛选参数值非法时返回统一错误结构
- `sourceUrl` 缺失或格式非法时阻止岗位发布
- 删除或下架后的岗位不再出现在公共列表

继续沿用现有统一响应格式：

```json
{
  "code": 200,
  "message": "success",
  "data": {}
}
```

为后续统一搜索保留接入位：

- 列表筛选参数命名保持稳定，后续可直接作为统一搜索的 jobs 子域过滤器。
- 岗位主表字段命名避免与搜索索引模型冲突。
- 首页和路由继续使用 `/jobs`，避免后续再做二次迁移。

## 11. 测试与验收

### 11.1 后端测试重点

- 岗位列表筛选
- 岗位详情可见性
- 岗位创建与编辑
- 岗位发布 / 下架 / 删除
- 岗位收藏幂等
- 个人岗位收藏查询
- 普通用户越权访问后台接口

### 11.2 前端测试重点

- 岗位列表页筛选与空态
- 岗位详情页跳转与收藏交互
- 管理员岗位维护页
- 首页 `jobs` 入口真实跳转
- 游客与登录用户的按钮差异

### 11.3 本机冒烟验收

至少验证以下链路：

1. 游客可浏览岗位列表和详情，并跳转外链。
2. 登录用户可收藏岗位，并在个人中心继续看到岗位收藏。
3. 管理员可创建岗位草稿并发布。
4. 管理员下架后，前台公共列表与详情不再可见该岗位。
5. 首页 `jobs` 入口可直接进入岗位列表页。

## 12. 实施顺序建议

建议按以下顺序实施：

1. 先冻结岗位数据模型与枚举
2. 再完成公共列表 / 详情 / 收藏接口
3. 再完成管理员维护接口
4. 再接岗位列表页、详情页和后台维护页
5. 最后接首页入口、个人中心收藏复用和完整联调
