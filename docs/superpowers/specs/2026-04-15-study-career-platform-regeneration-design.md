### 2.2 新项目边界

新版项目不是泛校园信息门户，而是聚焦大学生三条升学就业路径的一站式平台：

- 就业信息聚合与经验分享
- 考研院校与资料支持
- 出国留学信息与对比决策

核心角色为：

- 游客
- 普通用户
- 已认证用户
- 管理员

2. `requirements/`：基于新版需求说明书，重新生成全局正式需求清单。
3. `plans/`：按子项目分阶段生成实施计划，而不是试图一次规划完整平台。

## 3. 全局拆分

### 3.1 模块域拆分

当前项目按以下业务域拆分：

| 域 | 说明 |
| --- | --- |
| 平台基础域 | 注册登录、JWT 会话、首页框架、个人中心、通知、基础权限 |
| 社区域 | 发帖、评论、点赞、热帖排行、经验帖推荐 |
| 信息聚合与资料域 | 岗位浏览筛选、资料上传下载审核、统一搜索 |
| 决策辅助域 | 路径推荐测评、备战时间线、院校对比 |
| 数据看板域 | 热度统计、趋势图、选择比例、运营分析 |
| 管理域 | 内容审核、认证审核、用户管理、运营后台 |

### 3.2 推荐实施顺序

推荐按以下顺序推进：

| 阶段 | 范围 | 目标 |
| --- | --- | --- |
| A | 用户与认证 + 首页框架 + 管理后台最小骨架 | 建立可登录、可审核、可进入后续模块的基础壳 |
| B | 社区域 | 建立内容生产与互动主链路 |
| C | 信息聚合与资料域 | 建立岗位、资料库和统一检索 |
| D | 决策辅助 + 数据看板 | 完成评估、对比和可视化能力 |

当前只对阶段 A 进入可规划级设计，B/C/D 仅保留边界和接入点，不在本次计划中展开。

## 4. 阶段 A 设计范围

### 4.1 阶段目标

阶段 A 的目标不是提前做完整业务，而是建立后续所有模块共享的基础骨架，使平台具备以下最小可运行能力：

- 游客可完成注册与登录
- 已登录用户可进入独立首页和个人中心
- 普通用户可提交学号认证
- 管理员可在后台审核认证申请
- 系统可向用户投递最小站内通知
- 前后台路由、权限和页面壳已具备后续扩展能力

### 4.2 本阶段必须包含

- 手机号 + 验证码注册登录
- JWT 登录态管理
- 游客、普通用户、已认证用户、管理员四类身份的前后台访问控制
- 个人中心基础页
- 学号认证申请、审核、结果通知
- 独立首页聚合展示
- 管理后台首页和待审核列表最小骨架
- 最小通知中心

### 4.3 本阶段明确不做

以下能力不在阶段 A 的实施范围内：

- 发帖、评论、点赞、热帖排行
- 岗位浏览与筛选
- 资料上传下载与资料审核
- 院校对比
- 路径推荐测评
- 运营趋势图、热词分析、下载排行等完整看板

这些能力的入口可以在首页和后台中预留，但不能在 A 阶段被写成已交付功能。

## 5. 阶段 A 的功能设计

### 5.1 注册与登录

系统应保留需求说明书中的短信验证码语义：

- 手机号必须为 11 位数字
- 验证码为 6 位数字
- 有效期 5 分钟
- 注册成功后直接进入登录态
- 登录成功后签发 JWT

本阶段允许采用开发态短信发送实现，即保留“发送验证码”业务语义，但使用 mock 发送通道完成开发和联调，不强制接入真实短信服务。

### 5.2 学号认证

普通用户在个人中心提交学号和姓名后，系统生成一条认证申请，状态进入“审核中”。

管理员在后台审核时：

- 通过：用户身份升级为“已认证”，并产生认证结果通知
- 驳回：用户回到“未认证”，并携带驳回原因通知用户

同一用户在已有待审核申请时不能重复提交。

### 5.3 独立首页聚合展示

首页不是简单跳转页，而是平台统一入口。在仅完成阶段 A 的前提下，首页至少承担以下功能：

- 显示用户当前身份与认证状态
- 展示平台核心入口卡片：社区、岗位、资料、测评、看板、后台
- 展示用户待处理事项，例如“去完成学号认证”
- 展示最近系统通知或欢迎信息
- 对尚未交付的模块使用“即将开放”或空态说明，而不是伪造业务数据

游客访问首页时，应以登录注册引导和平台能力介绍为主；已登录用户访问首页时，应以导航聚合和个人状态提示为主。
首页入口应按权限显示，其中管理后台入口仅对管理员展示。
阶段 A 的“最近系统通知或欢迎信息”允许来自静态配置或初始化数据，不要求同时交付完整公告发布能力。

### 5.4 个人中心与通知

阶段 A 的个人中心至少包含：

- 基本资料查看
- 认证状态查看
- 学号认证提交入口
- 通知入口

阶段 A 的通知中心仅覆盖当前已落地事件：

- 欢迎通知
- 认证通过通知
- 认证驳回通知

通知需要支持未读数、已读标记和列表查看，但不要求在 A 阶段实现完整通知分类体系。

### 5.5 管理后台最小骨架

管理后台本阶段只承担“最小可运营”职责：

- 后台首页展示待审核认证数等待办摘要
- 审核列表支持查看申请详情
- 审核动作支持通过 / 驳回
- 驳回必须填写原因

后台导航结构应预留后续扩展位，用于接入帖子审核、资料审核、用户管理和运营数据，但这些功能在 A 阶段只保留菜单骨架或占位页。
后台首页的待办统计在 A 阶段只统计学号认证申请，不提前承接帖子、资料等未来审核数据。

## 6. 权限与边界

| 能力 | 游客 | 普通用户 | 已认证用户 | 管理员 |
| --- | --- | --- | --- | --- |
| 浏览首页与公开入口 | 是 | 是 | 是 | 是 |
| 注册 / 登录 | 是 | 否 | 否 | 否 |
| 进入个人中心 | 否 | 是 | 是 | 是 |
| 提交学号认证 | 否 | 是 | 否 | 否 |
| 查看认证结果通知 | 否 | 是 | 是 | 是 |
| 进入管理后台 | 否 | 否 | 否 | 是 |
| 审核认证申请 | 否 | 否 | 否 | 是 |

管理员同时拥有前台登录身份，但后台能力必须经过独立权限校验，不能仅依赖前端菜单隐藏。

## 7. 业务流与数据流

### 7.1 注册登录流

1. 游客输入手机号并请求验证码。
2. 系统校验手机号格式并生成验证码。
3. 开发态短信通道发送或记录验证码。
4. 用户提交验证码完成注册或登录。
5. 系统签发 JWT，返回用户身份信息与首页跳转上下文。

### 7.2 学号认证流

1. 普通用户在个人中心填写姓名与学号。
2. 系统创建认证申请，状态为“审核中”。
3. 管理员在后台查看申请并执行通过或驳回。
4. 系统更新用户认证状态。
5. 系统生成通知并更新用户未读数。

### 7.3 首页聚合流

1. 用户进入首页。
2. 系统读取用户身份、认证状态、未读通知数和模块入口配置。
3. 系统返回首页所需的聚合数据视图。
4. 前端按当前身份渲染快捷入口、提示信息和占位模块。

### 7.4 后台待办流

1. 管理员进入后台首页。
2. 系统读取待审核认证数量和最近申请列表。
3. 管理员进入审核页处理单条申请。
4. 处理结果回写申请状态并触发通知。

## 8. 架构设计

### 8.1 技术边界

整体仍遵循需求说明书中的既定技术方向：

- 后端：Java 17 + Spring Boot 3
- 前端：Vue 3 Web 应用
- 认证：JWT
- 部署：Docker Compose
- 核心基础设施目标：MySQL、Redis、Elasticsearch、MinIO、Nginx

阶段 A 只要求实现当前范围真正需要的能力，但代码和配置结构需要为后续 Redis、Elasticsearch、MinIO 接入保留清晰扩展位，避免 A 阶段做成一次性原型。
阶段 A 的实施计划可以不接入 Elasticsearch 与 MinIO 主业务链路，但不得破坏后续接入这些组件的配置与模块边界。

### 8.2 前端模块边界

前端按以下边界组织：

- 认证模块：注册、登录、登录态恢复
- 公共首页模块：游客首页、登录后首页、入口聚合
- 用户模块：个人中心、认证申请、通知列表
- 管理模块：后台布局、待办摘要、认证审核页
- 权限路由层：前台鉴权、后台鉴权、未授权跳转

### 8.3 后端模块边界

后端按以下服务职责划分：

- `auth`：验证码、注册、登录、JWT 签发与校验
- `user`：用户资料、角色、状态查询
- `verification`：学号认证申请与审核
- `notification`：通知创建、未读数、已读处理
- `home`：首页聚合查询接口
- `admin`：后台待办汇总与认证审核接口

这里的 `admin` 只承担后台聚合与审核入口职责，不提前承接未来所有后台业务逻辑。

### 8.4 数据对象建议

阶段 A 建议至少明确以下数据对象：

| 对象 | 关键字段 | 用途 |
| --- | --- | --- |
| User | id, phone, nickname, role, status, student_id, is_verified, created_at | 用户基础信息与身份状态 |
| StudentVerificationApplication | id, user_id, student_id, real_name, status, reject_reason, reviewer_id, reviewed_at, created_at | 认证申请与审核结果 |
| Notification | id, user_id, type, content, is_read, source_id, created_at | 最小站内通知 |

验证码可使用专门存储或缓存机制承载，但对外业务语义应固定为“手机号验证码验证”，不能把 mock 实现直接暴露为业务设计。

## 9. 错误处理与降级

阶段 A 至少需要覆盖以下异常场景：

- 手机号格式错误：阻止发送验证码并返回明确提示
- 验证码错误或过期：注册 / 登录失败并提示重试
- Token 失效：前端清理登录态并跳转登录页
- 重复提交认证申请：返回“已有待审核申请”
- 管理员驳回未填写原因：阻止提交
- 数据库异常：返回统一错误响应并记录日志

A 阶段虽然尚未接入搜索和对象存储主链路，但错误响应格式必须与全局要求一致，即 `{code, message, data}`。

## 10. 测试与验收

### 10.1 测试重点

- 注册、登录、登出、Token 续航与失效跳转
- 验证码过期、重复发送、重复注册
- 认证申请提交流程
- 管理员通过 / 驳回流程
- 通知生成、未读数更新、已读状态变更
- 游客、普通用户、已认证用户、管理员的页面访问边界
- 首页在游客与登录用户两种身份下的聚合展示

### 10.2 阶段 A 验收标准

当满足以下条件时，可判定阶段 A 设计进入可实施状态：

- 游客能通过手机号验证码完成注册和登录
- 登录用户能进入独立首页和个人中心
- 普通用户能提交学号认证申请
- 管理员能在后台审核认证申请并产生结果通知
- 首页能够聚合展示用户状态、入口卡片和最小通知
- 后台具备后续扩展结构，但未交付模块不会被误认为已完成
- 全部接口保持统一响应结构和明确权限边界
